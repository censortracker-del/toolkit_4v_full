# Agent Wiki Master Playbook — MAX (protocol_version 4, universal single-paste build)

Self-contained rescue prompt + operating manual for a lane-based multi-agent workflow (User + implement lane + review lane + audit lane). Domain-agnostic: codebases, websites, research, datasets, content libraries, business docs, analytics, legal/registry material, product planning. Supersedes `PRINCIPLES_MAX.md` v3. Daily use = `PRINCIPLES_MINI.md`; this MAX file = recovery / bootstrap / single-paste deployment reference.

`protocol_version: 4`

**This file is self-sufficient.** Pasting this one file is enough to deploy a working Agent Wiki on any machine, with no repository and no second file. Literal copy-and-run starter files live in **Appendix A**; the domain adapter (one filled reference + one blank + adaptation rules + interview checklist) lives in **Appendix B**.

**Honest boundaries.** Discipline, not a guarantee: it saves tokens by routing agents to small deltas instead of re-reading whole projects, but it can only DETECT non-compliance after the fact (stale cursor vs newer `last_change_id`), not FORCE it. It protects against accidental re-reading, not an agent that ignores the rules. Same-engine lanes are weak cross-validation; real independence = the human + a different engine covering at least one lane.

**v4 note.** v4 renames all agent-facing artifacts to lane-neutral form (Review_Inbox, Audit_Inbox, _review_cursor, _audit_cursor, AGENT_CONTEXT, needs_review/needs_audit) and drops all vendor role names from the core: roles are Implementer / Reviewer / Auditor, bound to concrete agents per project via `lane_binding` in `Agent_Protocol.md`. Domain legacy is fully removed from the toolkit; v3 (with vendor names and book-project notes) is frozen at git tag `v3.2`. A v3 project keeps working as-is; to migrate, apply the rename map from `MANIFEST.json` and bump `protocol_version` to 4.

**Amendments.** 2026-06-29: role_mode (SOLO/DUAL/FULL) added (v3.2). 2026-07-04: v4 — lane-neutral naming; task_type axis + TASK tier (ROUTER v3); core purity rule; protocol_version 4. Core mechanics (lanes, cursors, entry_id, AGENT_CONTEXT, events, safety) unchanged.

## 1. Core philosophy

Chat history is fragile memory: it truncates, repeats, wastes tokens, forces re-deriving state. Use a local **wiki as durable memory** — small Markdown/TSV files (short header, index, change log, cursors, inboxes, reports, optional item pages). Read the header first; stop if nothing changed; otherwise open only the listed scope. This saves re-reading *state*, not the reasoning itself (ingest/content review cost what they cost). The human curates and decides; agents do analysis, implementation, review, bookkeeping.

## 2. Three layers

* **Raw sources (immutable, source of truth):** repo / documents / database / dataset. Read-only to reasoning agents; no move/rename/delete/mutate without explicit user confirmation + dry-run. Not the place for agent memory.
* **Wiki (LLM-owned):** Implementer generates; everyone reads; Reviewer/Auditor write only to their inboxes + cursors; generated blocks regenerated by Implementer.
* **Schema/config:** this file + project-local `Agent_Protocol.md` (the domain adapter) + role defs + prompts + index columns + safety rules. Evolves via the change log (`entry_id`), not chat.

## 3. Four roles

* **User** — owner/decider: direction, naming, permissions, money/API, deletion approval, human-only tradeoffs, "good enough".
* **Reviewer** — architect, semantic reviewer, sign-off: requirements, architecture/product, decomposition, content/recommendation review, decides design questions Auditor flags, **decides contradiction resolution and judges wiki semantic health (lint)**. Writes only `Review_Inbox/feedback_N.md` + `_review_cursor.txt`. No code.
* **Implementer** — builder/data operator: code, scripts, tests, DB, exports, reports, wiki generation, dry-runs, accepted fixes, updating generated context/index/log/cursor, **runs lint mechanics and detects/flags contradictions (does not resolve them)**. Owns generated files. No silent destructive actions.
* **Auditor** — read-only code + safety + data-integrity auditor. Writes only `Audit_Inbox/audit_N.md`, `deal_N.md`, `_audit_cursor.txt`. Flags architecture; Reviewer decides; Implementer implements.

**Lane discipline:** flag outside your lane, decide only inside it. User overrides anyone.

**Deployment configurations (role_mode, set at bootstrap by ROUTER; role/lane
definitions stay fixed).** role_mode is about LANE coverage, not vendor names. The
three lanes are: implement, review/semantic, audit/safety. Deploy the subset the
project needs:

* **SOLO (1 agent):** one agent covers all three lanes — implements, then reviews
  its own work, then audits it for safety/data-integrity, then bookkeeps. Lane
  discipline still applies *internally* (separate "I am implementing" from "I am
  auditing this"), but per honest boundaries self-review is the weakest
  cross-validation. No separate inboxes/cursors needed; the single agent owns the
  generated files. Use when independence need is low.
* **DUAL (2 agents):** an implementer (implement lane) + one reviewer (review +
  audit lanes folded together). Two cursors optional; one inbox for the reviewer.
* **FULL (4 agents):** User + the three lanes as separate agents, as defined above.
  Use when real independence matters (a different engine as auditor) and damage from
  an uncaught mistake is high.

Lanes bind to whatever agents actually run, recorded per project as `lane_binding`
in `Agent_Protocol.md` — e.g. `implement=Codex, review=Claude, audit=OpenCode`, or a
local-model binding `implement=Assembler, review/audit=local-reviewer` (Ollama /
LM Studio). Same lanes, different agents; core files never depend on the binding.

Tier and role_mode are two independent axes: a project can be MAX-tier (rigorous
bookkeeping) yet SOLO (one agent). ROUTER sets both.

(Project-specific legacy name maps, if any, belong in that project's own notes file, not here.)

## 5. Entry point — 00_START_HERE.md

Top of the file is an Implementer-owned `AGENT_CONTEXT` block — the cheapest thing every agent reads first. Its fields and what they mean:

* `protocol_version` — lets an agent detect an old schema.
* `updated` — ISO timestamp of last regeneration.
* `project_run` — run/session id.
* `last_change_id` — newest `entry_id` in the change log; the value cursors compare against.
* `needs_review` / `needs_audit` — `yes`/`no`.
* `review_scope` / `audit_scope` — exact files to open if review is needed, else `—`.
* `counts` — domain counts (e.g. `items=0`).

Below it: a `PROTOCOL_LINK` block (protocol + cursor pointers) and an Implementer-owned `WIKI` block (generated nav). **The literal copy-and-run template for this file is in Appendix A** — that is the single source of truth; do not maintain a second literal copy elsewhere.

## 6. Index — Project/_index.tsv

Cheapest overview: one tab-separated row per item; **compact and routing-only**. Domain columns + the agreed status fields:

```tsv
id	title	...domain...	status	needs_manual_review	last_processed_change_id
```

`confidence` only if actively used for routing/triage. Keep detailed fields (source_path, generated_path, last_verified_at, freshness, notes, evidence) on item pages / SQLite / reports — NOT in the index, or it stops being cheap. Read `_index.tsv` before item pages.
Suggested `status` values: raw, parsed, classified, review_needed, approved, exported, archive_candidate, delete_candidate, error. `archive_candidate`/`delete_candidate` are recommendations only.

## 7. Change log — Project/Change_Log.md

Append-only, newest on top. Each significant change: stable unique `entry_id` + `changed_files` (so reviewers can check scope) + added/removed/status_changes/significant + review_flags. Reference by `entry_id`; link files, don't paste diffs.

## 8. Error log — Project/Error_Log.md

Lightweight, human-readable, **critical / manual_review failures only**. If structured errors already live in SQLite, this file is GENERATED from SQLite — it is not a competing source of truth. Per-entry: error_id, item_id, source_path, stage, agent, change_id, error_type, message, retry_possible, needs_manual_review, suggested_action.

## 9. Cursors + compliance detection (NOT enforcement)

Each reviewer owns one cursor (last processed `entry_id`). At start, write `started_at`, `last_change_id`, `read_context: yes/no` to cursor/log — an **audit trail**, not enforcement: a session that skips `AGENT_CONTEXT` also skips this line; the only signal is a stale cursor vs a newer `last_change_id`. If a reviewer can't write its cursor, Implementer updates it after processing.

## 10. Review-trigger checklist (how Implementer sets flags)

A raw diff can't decide if review is needed (a comment vs removing `con.commit()` both touch a `.py`) — judgment call:

* SQL / writes / saving / file output / DB schema → `needs_audit: yes` + scope.
* architecture / data model / report format / recommendation logic → `needs_review: yes` + scope.
* contradiction detected during ingest → flag it (see §13) + `needs_review: yes` + scope (the conflicting pages).
* comments / renames / formatting / test-only → none.
* unsure → link diff, let reviewers decide.

## 11. Inboxes / feedback / reviews / deals

`Review_Inbox/feedback_N.md`; `Audit_Inbox/audit_N.md` + `deal_N.md`. Highest N newest, never overwrite. Deals group related open questions tagged by addressee (User/Reviewer/Implementer/both); one deal = one cluster; closes when all resolved; answers in chat or a response file.

## 12. Agent loops (run WHEN INVOKED — semi-automatic)

**Review/Audit lane (Reviewer or Auditor):** read header → read cursor → if `last_change_id`==cursor stop → if your lane flag (`needs_review` / `needs_audit`)==no stop → else open ONLY your `*_scope` → review → write inbox file → update cursor.
**Implementer:** read request + entry point + relevant files → implement/inspect (no destructive action without dry-run + user confirm) → verify → regenerate outputs → update index/header/log if significant → set flags via §10 → tell user the exact path or next action.
**START checklist (every agent):** I read 00_START_HERE; read my cursor; compared cursor vs last_change_id; know my role/lane; know read-only vs write; know whether user confirmation is required.

## 13. Event types

**Ingest** → Implementer processes a new source, updates index/items/reports, appends log, sets flags. **Inference at ingest:** any classification, typing, categorization, risk/scenario assignment, or other derivation produced while ingesting is stored as `agent_inference` (with `confidence` + `needs_manual_review`), linked to its supporting `source_evidence` — never written as source fact (see §16). **Contradiction handling:** if the new source conflicts with an existing wiki claim, do NOT silently overwrite — Implementer flags the contradiction, links both sources, marks the affected page (and `needs_manual_review` if warranted), sets `needs_review` + scope, and logs the conflict; Reviewer decides the resolution.

**Code change** → edit, test, log, flag if §10.

**Architecture/semantics** → Reviewer reviews/proposes, Implementer implements, log, flag.

**Query** → an agent answers a user question from the `Agent_Wiki` (and raw-source scope), with citations to the pages used. **File-back:** if the answer produces reusable synthesis, a comparison, a decision, or a finding, file it back into the wiki as a new page/report and log it — explorations should compound in memory like ingested sources, not vanish into chat. **Provenance:** a Query-filed page is `agent_inference` (synthesis/derivation), not source fact — mark it as such on the page and link the source pages it rests on (see §16). Never let a derived synthesis become indistinguishable from a source claim.

**Lint** → periodic wiki health-check. **Implementer runs the mechanical pass** (orphan pages with no inbound links, broken/missing cross-references, stale-by-date claims, duplicated pages, unresolved `manual_review` items, data gaps a web check could fill) and produces a lint report. **Reviewer judges the semantic issues** (contradictions between pages, weak or outdated synthesis, which of two conflicting claims is authoritative, concepts mentioned but lacking their own page). Mechanical detection ≠ semantic decision: findings that need a meaning judgment go to Reviewer. Lint is run when invoked, not autonomously.

**Destructive action** → dry-run manifest → user reviews → explicit confirm → execute only approved manifest → rollback/log. Never skip dry-run.

## 14. Token economy

Header first; `_index.tsv` before item pages; reference by id, don't re-quote; open exact `*_scope`; don't scan all inboxes or old feedback by default; short but complete, no hard char caps; durable summaries into the wiki; chat is conversation, not memory.

## 15. Safety rules

* No direct move/rename/delete of source files — dry-run manifest + user review + explicit confirmation + execution log + rollback if feasible.
* **Preserve user-provided content by default:** add, clarify, annotate, or flag — never delete/overwrite user content unless the user explicitly requests it.
* No secrets in any file/log/DB/export/prompt/screenshot/error.
* No automatic paid/large LLM runs on file change; user confirms.
* Labels (delete_candidate/archive/deprecated) are recommendations, never deletion permission.
* Unknown / weak-evidence / scan-only / OCR-bad / ambiguous / low-confidence → manual_review; not exported as confirmed.
* Never hand-edit generated blocks (WIKI / AGENT_CONTEXT / PROTOCOL_LINK); Implementer regenerates them.

## 16. Evidence & classification discipline

* **Provenance:** keep `source_evidence`, `agent_inference`, `external_evidence`, `user_override` separate — never merge into one undifferentiated field. Carry `confidence` + `needs_manual_review`. Anything the agent derives — a classification or risk/scenario assignment at ingest, or a synthesis/comparison/finding filed by a **Query** — is `agent_inference`: mark it as agent-derived and link the source pages/evidence it rests on.
* **No silent normalization:** raw source strings (author/title/name) are evidence. Store normalization only as a separate candidate/hypothesis (`raw_*` vs `normalized_*_candidate` + `normalization_confidence`), and only if the project allows it.
* **No invention:** never present a value not present in the source as `source_evidence`. If a code/relation/source/factual field is absent, leave it empty or mark unknown/manual_review. If classification, interpretation, recommendation, risk assessment, or other inference IS the task, output it as `agent_inference` with `confidence` + `needs_manual_review` and links to supporting `source_evidence` — never present it as `source_evidence`.
* **Contradiction = flag, not overwrite:** when sources or pages disagree, record both with provenance and route to Reviewer; do not silently pick one.
* **Internet checks** (when run) record: `checked_at`, `source_url`, `source_title`, `source_date`, `claim_supported`, `claim_confidence`; for volatile topics add `freshness_sensitivity`, `recheck_after`.

## 17. Data-integrity patterns

```python
# BAD: all in memory, single final commit — crash loses everything
rows = [process(x) for x in items]; db.executemany("INSERT ...", rows)

# GOOD: raw-to-disk before parse, checkpoint per unit, commit per batch
for x in items:
    raw = save_raw_response(x)      # 1) raw response to disk first
    r = parse(raw)                  # 2) parse from saved copy
    upsert_one(con, r)              # 3) idempotent upsert on a stable id
    con.commit()                    #    commit per row/batch

tmp = final + ".tmp"; write(tmp, data); os.replace(tmp, final)   # atomic replace
```

Use transactions + unique constraints; idempotent reruns; checkpoint to resume; record parse errors to Error_Log/SQLite; never lose expensive API work on interruption.

## 18. Completion criteria

Done only when: work done; outputs regenerated; tests/checks run or limits stated; counts/index/context reconcile; significant change logged with `entry_id` + `changed_files`; review flags accurate; Query findings filed back if reusable; user told the exact output path or next action. NOT done: "changed something" with no path; "should work" unverified; "delete candidates" without dry-run + confirm; "review needed" without scope; a reusable Query answer left only in chat.

## 19. Bootstrap a new project (deployment)

**Starter source rule (honest, ordered):**

1. If an external `STARTER_PACK.md` or starter repository is present in context → use it.
2. Else → use **Appendix A** inside this file (the single-paste path).
3. Else (neither available) → ask the user for the starter skeletons; do NOT improvise file formats.

**Steps:** Copy the starter (Appendix A or external). Then run the domain adapter (**Appendix B**):

* If the domain matches a known filled adapter (Appendix B1) → use it as the form template.
* If the domain is new → run the **interview checklist (Appendix B3)** to fill every adapter slot before generating files. Do not guess slot values.

Decide and record in `Agent_Protocol.md`: `lane_binding` (which concrete agent covers implement / review / audit — per role_mode; one agent may cover several lanes); source-of-truth path; `item` definition; `_index.tsv` columns; `counts`; lifecycle statuses; domain safety rules; output types; Reviewer vs Auditor review triggers; language conventions (user-facing RU / agent-facing EN).

Then: create initial `Change_Log.md` + cursors (= initial `entry_id` `e0001`, identical across both cursors and `last_change_id`); generate initial index; flags = no; tell the user the file to open. The starter ships copy-and-run skeletons (Appendix A): `00_START_HERE.md`, `_index.tsv`, `Change_Log.md`, both cursors, `Agent_Protocol.md`, `Review_Inbox/README.md`, `Audit_Inbox/README.md`. `Error_Log.md` is NOT in the starter — Implementer creates it on the first critical/manual_review error, or generates it from SQLite if structured errors live there.

## 20. Recovery if context lost

0. Validate the `AGENT_CONTEXT` block parses; if corrupt, fall back to the newest `entry_id` in the log.
1. Open 00_START_HERE (top blocks) → Agent_Protocol → Change_Log → _index.tsv → cursors → continue from `last_change_id`.
2. If files don't exist: use this MAX file, recreate the starter **from Appendix A**, ask the user for the source path, run the Appendix B adapter (interview if domain is new), build initial index + log, proceed.

## 21. Customize vs fixed

Customize: `lane_binding` (which agent covers which lane); source-of-truth layer; `_index.tsv` columns; `counts`; domain safety rules; reports/item pages; output types; language conventions; deployment configuration (role_mode SOLO/DUAL/FULL, set by ROUTER). Fixed: role/lane definitions; inbox naming; cursor mechanism; `entry_id`; `AGENT_CONTEXT`; exact review scopes; dry-run before destructive actions; no secrets; no automatic expensive runs; generated blocks owned by Implementer; `protocol_version`; the core event set including **Query, Lint, and contradiction handling**.

## 22. Deferred (NOT adopted in v4 core)

Revisit only for a larger/different-domain project: `PROJECT_RULES.md`, `Project/dry_run_manifest.tsv` as a standing file, `File_Naming_Policy.md`, `Export_Contracts/`, `maturity_level` axis, the full 12-column `_index.tsv`, and a standalone CLI search engine over wiki pages (e.g. BM25/vector) once the index file stops scaling.

## 23. Final principle

The wiki is the shared brain. The chat is the hands. The user is the owner. The Reviewer decides meaning. The Implementer changes the system. The Auditor guards against damage. Everything important becomes a file with a path.

## B3. Adaptation rules + interview checklist

**One slot list, two read modes.** Known domain → use the *structure/shape* of the filled reference adapter (B1) as the form, NOT its domain-specific values. New/unknown domain → ask the interview checklist and fill each slot from the user's answers; never copy B1 demo values. Either way every slot below must be answered before bootstrap — no guessing, no improvising.

> **B1 is a reference of adapter shape, not a universal default domain profile.** Do not copy its demo paths, statuses, safety rules, counts, or output types into any real project unless the user explicitly confirms they fit.

Slots / interview questions:

1. **Source of truth** — where do raw sources live, and is it confirmed read-only to reasoning agents?
2. **Item definition** — what counts as one "item"? What stable id identifies it (and is it path-independent)?
3. **Index columns** — which few routing-only columns go in `_index.tsv`? (Detailed fields stay on item pages / DB / reports.)
4. **Counts** — which counts does `AGENT_CONTEXT` track for this domain?
5. **Lifecycle statuses** — which `status` values describe an item's life from raw to exported?
6. **Domain safety rules** — what must never be moved/renamed/deleted/exported? What triggers manual_review here?
7. **Output types** — markdown pages? reports? tables? Marp slides? charts? Which does this domain need?
8. **Reviewer review triggers** — which changes are semantic/architecture/recommendation decisions (Reviewer's lane)?
9. **Auditor review triggers** — which changes touch code/writes/schema/data-integrity (Auditor's lane)?
10. **Language conventions** — user-facing language (RU/EN); agent-facing files stay EN.
11. **Linked project notes** — does this project carry any legacy/notes file that must stay reachable from `00_START_HERE`?
12. **lane_binding** — which concrete agent covers each lane (implement / review / audit), per the role_mode ROUTER selected? One agent may cover several lanes (SOLO/DUAL).

Rules:
* Fill all 12 before creating files. Missing answer → ask, don't invent.
* As you instantiate new domains, save each filled adapter into the toolkit's `adapters/` directory as `adapters/ADAPTER_<domain>.md` — as a PROPOSAL the user approves (core purity: agents never silently edit the toolkit). Do NOT inline domain adapters into this core body; the body holds only the toy B1 demo as the canonical form example.
* The filled adapter becomes the project's `Agent_Protocol.md` (Appendix A8), the project-local schema.

**Flag closure rule:** any raised review/audit flag may only be closed as `satisfied` (verdict file in the inbox) or `waived_by_user` (explicit user decision recorded in Change_Log). Silently dropping a flag is a protocol violation.
